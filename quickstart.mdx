This guide walks you through installing Spectre, configuring permissions, and verifying that it's actively protecting your server.

## Prerequisites

Before you begin, ensure you have:
- A running Hytale server (version 2026.02.06 or later)
- Spectre installed (see [Installation](/installation) if you haven't already)
- Server operator or admin permissions

## Step 1: Verify Spectre is loaded

<Steps>
  <Step title="Start your server">
    Launch your Hytale server with Spectre in the mods folder.
  </Step>

  <Step title="Check the server logs">
    Look for this message in your server console:

    ```
    [INFO] Setting up Anticheat
    ```

    This confirms Spectre loaded successfully and registered its checks.
  </Step>
</Steps>

<Info>
If you don't see the setup message, verify that the Spectre JAR file is in your server's `mods/` directory and restart the server.
</Info>

## Step 2: Configure permissions

Spectre uses two permission nodes to control who can bypass checks and who receives alerts.

<Steps>
  <Step title="Grant bypass permission to staff">
    Staff members and operators should be exempt from anticheat checks. Grant them the bypass permission:

    ```
    solarion.anticheat.bypass
    ```

    Players with this permission will not be checked by any Spectre systems.
  </Step>

  <Step title="Enable alert notifications">
    To receive real-time violation alerts, grant staff the alerts permission:

    ```
    solarion.anticheat.alerts
    ```

    Staff with this permission will see chat messages when players fail checks.
  </Step>
</Steps>

<Warning>
Creative mode players are automatically bypassed regardless of permissions. This prevents false positives from legitimate creative flight.
</Warning>

## Step 3: Test violation detection

Now let's verify Spectre is actively monitoring players.

<Steps>
  <Step title="Join the server as a non-staff player">
    Connect to your server with a regular player account (one without `solarion.anticheat.bypass`).
  </Step>

  <Step title="Monitor the console">
    Watch your server console for violation messages. Normal gameplay should not trigger any warnings.

    If Spectre detects suspicious behavior, you'll see messages like:

    ```
    [WARNING] [Spectre] PlayerName failed Speed (VL: 2.0)
    ```
  </Step>

  <Step title="Observe automatic enforcement">
    If a player's violation level (VL) reaches **20.0**, Spectre automatically kicks them:

    ```
    PlayerName was kicked: Kicked by Spectre: Speed
    ```

    The player is disconnected and their VL is reset.
  </Step>
</Steps>

<Tip>
VL decays naturally at **0.05 per tick** when players behave normally. This means minor violations from lag or edge cases won't result in kicks unless they're repeated frequently.
</Tip>

## Understanding violation levels

Spectre tracks violation levels (VL) per player. Each failed check adds a weighted amount to the player's VL:

| Check | VL Weight | Description |
|-------|-----------|-------------|
| Flight | 10.0 | Instant ban threshold—flying without permission |
| Mantle | 5.0 | Severe exploit—invalid climbing state |
| Relative Speed | 3.0 | Moderate—excessive relative movement |
| Packet Rate | 3.0 | Moderate—packet spam attack |
| Speed | 2.0 | Minor—may be lag or edge case |

**Kick threshold**: When a player's total VL reaches **20.0**, they are automatically kicked.

## What happens when a check fails

When Spectre detects a violation:

1. **VL is incremented** by the check's weight
2. **Staff are alerted** (if they have `solarion.anticheat.alerts` permission)
3. **Movement is corrected** — the player is teleported back to their last valid position
4. **If VL ≥ 20.0** — the player is kicked from the server

## Monitoring alerts

Staff members with the `solarion.anticheat.alerts` permission receive throttled chat messages when violations occur:

```
[Spectre] PlayerName failed Speed (VL: 2.0)
```

Alerts are throttled to **once every 2 seconds per player** to prevent chat spam.

<Note>
Alert messages are only sent to online staff members with the alerts permission. They are not logged to files or persisted.
</Note>
